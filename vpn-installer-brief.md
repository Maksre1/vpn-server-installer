# Задание: скрипт автоматической настройки VPN-сервера на VPS

## Контекст и цель

Нужен open-source bash-скрипт (публикуется на GitHub, устанавливается через `curl | bash`), который на чистом VPS полностью настраивает рабочий VPN-сервер с несколькими протоколами, WARP-туннелированием и веб-панелью управления — без необходимости иметь собственный домен.

**Твоя роль:** самостоятельно найти актуальную документацию и лучшие практики по всем перечисленным ниже компонентам (версии, флаги установки, форматы конфигов могли измениться — не полагайся на память, проверяй), спроектировать и написать полный код проекта, подготовить структуру репозитория и README, и выложить как готовый к использованию проект. Это не пошаговый сценарий диалога — прими архитектурные решения сам там, где ниже сказано "на твоё усмотрение", и обоснуй выбор в README.

## Целевая аудитория и требования к качеству

Скрипт будет запускать множество разных людей на самых разных VPS (разные провайдеры, ядра, объём RAM, виртуализация). Требования:
- Идемпотентность — повторный запуск не должен ломать рабочую систему
- Явный error handling на каждом шаге, понятные сообщения об ошибках
- Graceful degradation — если какой-то шаг не удался (например, не выдался сертификат), скрипт не должен падать молча или ломать уже настроенное; нужен fallback и понятное предупреждение в финальном отчёте
- Все секреты (пароли, ключи, UUID) генерируются криптостойким рандомом (`openssl rand`), не `$RANDOM`
- Подробный лог установки в файл, для отладки и issue-репортов на GitHub

## Режимы работы

Скрипт при запуске спрашивает **ровно один вопрос**: Auto или Manual (дефолт Auto, автостарт через ~10 сек, если пользователь не ответил).

### Auto-режим — упор на пользователей без собственного домена

Ноль дальнейших вопросов. Полный автономный флоу:

1. **Пре-флайт проверки:** root-права, ОС/версия, архитектура (amd64/arm64), виртуализация (`systemd-detect-virt`, пропускать несовместимые sysctl-тюны с warning для OpenVZ/LXC), версия ядра, объём RAM/диска (создать swap при необходимости на слабых VPS), занятость портов, публичный IP (v4 обязателен, v6 опционально)

2. **Домен и TLS через sslip.io:**
   - Сформировать домен вида `IP-с-дефисами.sslip.io` (важно: дефисы, не точки — точки создают паразитные поддомены)
   - Проверить, что домен резолвится в свой же IP, прежде чем идти дальше
   - Fallback-цепочка wildcard-DNS сервисов на случай недоступности sslip.io: `sslip.io → nip.io → traefik.me` (проверь актуальность и надёжность каждого на момент разработки, возможно есть более свежие/стабильные аналоги)
   - Получение сертификата через acme.sh или certbot, HTTP-01 challenge (временно открыть порт 80, закрыть после)
   - Если все DNS-сервисы и/или Let's Encrypt недоступны — fallback на self-signed сертификат, с явной пометкой деградации в финальном отчёте и в панели (некоторые протоколы, в первую очередь NaiveProxy, заметно теряют в маскировке без настоящего TLS)
   - Настроить автопродление сертификата (systemd timer/cron)

3. **Тюнинг системы:** BBR (проверить доступность BBRv3 на новых ядрах, fallback на BBRv1/cubic), fq/fq_codel qdisc, tcp_fastopen, увеличение сетевых буферов, блокировка ICMP через iptables/nftables (определи, что сейчас стандартнее и стабильнее для целевых дистрибутивов)

4. **Протоколы — все четыре сразу, без вопросов:**
   - Hysteria2
   - Mieru
   - AnyTLS
   - NaiveProxy

   Для каждого: официальный способ установки (бинарник/репозиторий, не компиляция на слабом VPS), рандомный порт с проверкой занятости через `ss`, рандомные надёжные пароли/UUID, systemd unit с автозапуском и автоперезапуском при падении.

   Все четыре протокола заявлены как поддерживаемые в Clash Verge Rev (mihomo core) — **обязательно перепроверь актуальный статус поддержки каждого протокола в текущей стабильной версии mihomo** перед тем как проектировать формат единого конфига. Если для какого-то протокола нужны нестандартные настройки в mihomo (специфичные поля, обязательные плагины и т.п.) — учти это в генераторе конфига.

5. **WARP:** установка через wgcf (или актуальный на сегодня способ — проверь, не устарел ли wgcf, возможно появились более простые методы), регистрация бесплатного аккаунта с retry/backoff при 429, весь трафик через WARP (без выбора режима — в auto только "всё через WARP"), самотест после поднятия (сверка внешнего IP до/после).

6. **Роутинг в клиентском конфиге:**
   - Встроенный список ru-доменов → DIRECT (заглушка на первое время, но заложи механизм лёгкой замены/обновления списка)
   - `GEOIP,RU,DIRECT` как fallback-правило на случай, если домен не попал в список
   - Всё остальное → PROXY (через WARP)

7. **Панель управления:**
   - Systemd unit, автозапуск, слушает на рандомном порту
   - Учётные данные по умолчанию `admin`/`admin`
   - **При первом входе — обязательный принудительный флоу смены пароля**: пользователь не должен иметь доступа к остальным разделам панели, пока не сменит пароль. Не просто плашка-напоминание, а блокирующий редирект
   - HTTPS на той же связке домен+сертификат, что и остальные сервисы
   - В панели: YAML-конфиг, тумблер "всё через WARP" / выбор конкретных доменов через WARP, возможность указать свои списки direct/proxy (по URL или вручную), просмотр статуса всех сервисов
   - Endpoint вида `https://домен:порт/случайный-путь/` — subscription-ссылка для Clash Verge

8. **Firewall:** открыть только необходимые порты (SSH, порты протоколов, порт панели), остальное закрыто по умолчанию

9. **Финальная самопроверка и вывод:** все systemd-сервисы active, WARP отвечает, сертификат валиден — чеклист в консоль + сохранённый файл с итоговыми данными доступа

### Manual-режим

Тот же флоу, но с вопросами перед выполнением:
- Свой домен (+ опционально Cloudflare API токен для DNS-01) или автоматический sslip.io
- Выбор протоколов из всех четырёх (чекбокс/по одному вопросу на каждый)
- Режим WARP: (1) весь трафик, (2) только выбранные домены/сервисы (например AI-сервисы — свой список, заглушка на первое время), (3) не использовать
- RU-домены: встроенный список / свой список по URL / отключить (всё через прокси)
- Настройка панели (порт, включить ли вообще)
- SSH-хардинг (опционально, дефолт — нет): смена порта, отключение password auth — с явным подтверждением и выводом нового порта перед применением, чтобы не отрезать пользователю доступ

## Что нужно исследовать перед началом кодирования

Не полагайся на то, что "и так известно" — многое в этой области меняется быстро:

1. **Hysteria2** — актуальный способ установки (официальный install-скрипт от проекта apernet/hysteria), актуальный формат конфига, рекомендации по маскировке/obfuscation
2. **Mieru** — официальный репозиторий и способ установки, актуальный формат конфига, поддержка в mihomo
3. **AnyTLS** — актуальный статус (это относительно новый протокол от автора sing-box) — уровень стабильности, актуальный способ разворачивания на сервере, реальный уровень поддержки в mihomo/Clash Verge на сегодня
4. **NaiveProxy** — актуальный способ установки (обычно как плагин к Caddy), требования к сертификату, поддержка в mihomo
5. **mihomo (Clash Verge Rev core)** — актуальная версия, подтверждение поддержки всех 4 протоколов, нюансы формата YAML для каждого
6. **wgcf / WARP** — актуален ли ещё wgcf как способ регистрации бесплатного WARP-аккаунта, или появились более современные альтернативы; текущие практики обхода rate-limit при регистрации
7. **acme.sh vs certbot** — что сейчас предпочтительнее для автоматизации в bash-скриптах, актуальный синтаксис HTTP-01 challenge
8. **sslip.io / nip.io / traefik.me** — текущая надёжность каждого сервиса, есть ли более новые/стабильные альтернативы wildcard-DNS
9. **BBR/BBRv3** — в каких ядрах доступен на сегодняшний день, актуальный синтаксис sysctl
10. Актуальные списки ru-доменов и AI-сервисов для заглушек (например, готовые community-поддерживаемые списки, которые можно использовать как основу или дефолтный источник)

## Дополнительные требования

- `uninstall.sh` — полный откат всех изменений
- Логирование каждого шага установки в `/var/log/vpn-installer.log`
- README на русском (и желательно английском) с честным описанием ограничений (self-signed fallback, зависимость от сторонних DNS-сервисов и т.п.)
- Многоархитектурность (amd64/arm64) при скачивании бинарников
- DNS для проксируемого трафика — предусмотри DoH/DoT в клиентском конфиге, чтобы избежать DNS-based блокировок на стороне провайдера пользователя
